1. Basic Concepts and Process Groups

  • group: A communication subgroup in distributed training.
  • global rank: Process index in the world communicator.
  • group rank: Rank index within the group (not always equal to global rank).
  • src/dst: The peer rank for send/recv; if group is set, it’s the group-local rank.

Example Group

If group = [2, 4, 6, 8, 10]:

2. Tensor Communication API

2.1 send / recv / isend / irecv

Argument Explanation

  • send(tensor, dst, group=None, tag=0)
    Send tensor to peer (group-local rank = dst).
  • recv(tensor, src, group=None, tag=0)
    Receive tensor from peer (group-local rank = src).
  • isend/irecv
    Asynchronous, returns a Work handle, call work.wait() to complete.

tag

  • tag is a message identifier; both send and recv must use the same tag to match.

group_dst/group_src

  • You rarely set these manually; framework infers from dst/src and group.

3. Object List Communication API

3.1 send_object_list / recv_object_list Usage

  • Used for sending/receiving Python object lists (arbitrary type); serialized under the hood.
  • Two-step protocol: send sizes first, then send serialized content.

3.4 Core Implementation Snippet

  • object_sizes_tensor: stores the size of each serialized object
  • object_tensor: one long tensor containing all serialized data
  • Slice and deserialize sequentially

3.5 About rank_objects

  • rank_objects is the return value of recv; it indicates which peer actually sent the message (generally equals src).
  • For multi-to-multi comms or src=ANY_SOURCE, this helps identify source, but does not affect deserialization.

4. group and src/dst Parameter Mapping

  • group: restricts communication domain; src/dst are local to group.

  • If group is set, src/dst must be group-local rank.

  • group_src/group_dst are inferred; rarely need to set.

  • Mapping:

    • global → group-local: group_ranks.index(global_rank)
    • group-local → global: group_ranks[group_rank]

5. Common Pitfalls

  • If group is set, src/dst must be group-local ranks, not global.
  • tag must match for send/recv pairs.
  • send_object_list/recv_object_list: object_list must have the same length/order.
  • group_src/group_dst almost never need to be set.

6. Summary

  • Tensor communication: send/recv/isend/irecv
  • Object communication: send_object_list/recv_object_list
  • group restricts comm domain, src/dst are group-local
  • tag matches send/recv pairs
  • rank_objects is just a return value from recv, not data content

7. Batch Point-to-Point: batch_isend_irecv

7.1 Introduction

torch.distributed.batch_isend_irecv supports batch launching of multiple isend/irecv ops for higher throughput (supports NCCL, Gloo, UCC backends).

API

  • p2p_op_list: list of torch.distributed.P2POp ops (each describes an isend/irecv).
  • Returns: list of Work handles (call .wait()).

7.2 Typical Scenarios

  • Pipeline communication, ring allreduce, high-throughput batch comms.
  • Supports mixed isend/irecv in one call.

7.3 API and Parameter Usage

Each P2POp defines one comm op:

  • op: dist.isend or dist.irecv
  • tensor: data tensor
  • peer: peer (group-local rank)
  • group: (optional) comm group
  • tag: (optional) message tag

7.6 Notes

Caution

  • For NCCL backend, set current device via torch.cuda.set_device!
  • For the first collective in a group, all group members must call this API, or behavior is undefined.
  • After first collective, partial group use is allowed.

7.7 Implementation Highlights

  • Backend may optimize via coalescing (NCCL).
  • Returns Work objects, call .wait() as needed.

8. Extended Summary

  • Tensor p2p comm: send/recv/isend/irecv/batch_isend_irecv
  • Object comm: send_object_list/recv_object_list
  • Batch p2p is best for pipeline/high-throughput comms
  • All APIs return Work handle (async/sync)
  • group/src/dst usage same as above

Hexo Deploy 自动部署：

每次更新内容后都要重新生成并上传文件，使用 Hexo 的部署功能可以简化流程。Hexo 支持多种部署方式，其中 Git 部署是常用方案之一。基本思路是利用 Git 把生成的静态文件推送到服务器或托管服务。概括了这种思路：在服务器上安装 Nginx 提供网页服务，用 Git 实现代码上传自动化，这样本地执行一次 hexo d（deploy）就能让网站更新。

提示： 使用 Git 自动部署需确保服务器开放 Git 所用的 SSH 端口（默认为 22），并配置好公钥免密登录，以便 Hexo 在本地能顺利推送到服务器。如果你的服务器 SSH 端口不是 22，可在部署配置中加入端口号或在 .ssh/config 中配置别名。对于不熟悉 Git 钩子的新手，也可以考虑使用简单的 rsync 脚本同步文件或借助 CI 平台实现部署，但原理类似。
